# par-term Sixel V1 Acceptance Criteria

## Purpose

Define the in-depth acceptance criteria for a Sixel-first v1 embedded graphics backend using `par-term-emu-core-rust`.

This document intentionally scopes v1 to what the prototype has evidence for:

```text
PTY child process
  → par-term terminal model
  → Sixel graphics decoded into TerminalGraphic RGBA
  → screenshot/artifact bridge
```

Kitty graphics remains v2. iTerm2 remains diagnostic. Bookokrat PDF/image remains unclaimed until a fixture proves graphics emission and capture.

## V1 product statement

V1 should be described as:

> Embedded par-term reader/process panes support text TUIs and Sixel-backed graphics artifacts. EPUB/text reading is supported. PDF/image rendering is supported only when the child process emits Sixel graphics or when a static artifact fallback is available.

Do not describe V1 as:

> Full terminal graphics support.

or:

> Bookokrat PDF/image support.

until those are proven.

## Required capabilities

A backend satisfying Sixel V1 may advertise:

```text
host_process = true
text_tui = true
embedded_pane = true
graphics_decode_sixel = true
graphics_render_artifact = true
resize_propagation = validated-or-degraded
mouse_passthrough = false-or-unknown
graphics_decode_kitty = false
graphics_decode_iterm = false
```

Capability language matters:

- `graphics_decode_sixel` means par-term captured a Sixel image into RGBA data.
- `graphics_render_artifact` means Omegon can produce a PNG/similar artifact from captured terminal state.
- It does **not** imply live inline image rendering in the active TUI.

## Acceptance gates

### Gate 1 — Probe validation stays green

Command:

```bash
cd .tmp/par-term-probe
cargo test
cargo run -- --validate --assert
```

Required checks:

- PTY smoke passes.
- Sixel capture passes.
- Sixel screenshot artifact passes.
- Direct Kitty parser check remains non-regressing.
- Bookokrat EPUB text capture passes if fixture is available.

Failure handling:

- Any hard validation failure blocks v1.
- Diagnostic Kitty/iTerm failures do not block v1.

### Gate 2 — Real Sixel fixture

Current synthetic Sixel is acceptable for proving parser plumbing, but V1 needs a fixture whose provenance is clear.

Required fixture:

```text
.tmp/par-term-probe/fixtures/tiny.sixel
```

Acceptance:

```bash
cargo run -- --sixel-fixture fixtures/tiny.sixel --assert
```

Expected:

```text
status=PASS
protocol=Sixel
pixels > 0
graphics_count >= 1
```

The fixture should be either:

- generated by a documented script/command; or
- checked in with a note explaining its source and expected dimensions.

### Gate 3 — Artifact screenshot bridge

V1 requires a reusable artifact bridge, not only the hardcoded Sixel screenshot path.

Required CLI shape:

```bash
cargo run -- --sixel --screenshot target/sixel.png --assert
cargo run -- --bookokrat cockpit-test-assets/pride-and-prejudice.epub --screenshot target/bookokrat-epub.png --assert
```

Acceptance:

- output file exists;
- output file is nonzero;
- `file target/name.png` identifies PNG;
- command output reports artifact path;
- screenshot generation failure is a structured failed check.

Minimum acceptable bridge:

```text
Terminal state screenshot → PNG artifact
```

Nice-to-have but not required for v1:

```text
Individual TerminalGraphic extraction → PNG artifact per graphic
```

### Gate 4 — Bookokrat EPUB text remains usable

Command:

```bash
cargo run -- --bookokrat cockpit-test-assets/pride-and-prejudice.epub --assert
```

Acceptance:

- Bookokrat launches under `PtySession`.
- Captured terminal text includes known document markers:
  - `Pride and Prejudice`; or
  - `Project Gutenberg`; or
  - `START OF`.
- Long-running TUI timeout is handled as expected, not as failure if content was captured.
- No graphics are required for EPUB/text pass.

### Gate 5 — PDF/image outcome classification

V1 does not require PDF/image success, but it does require honest classification.

Required command shape:

```bash
cargo run -- --bookokrat-pdf cockpit-test-assets/dummy.pdf
cargo run -- --bookokrat-pdf cockpit-test-assets/image-page.pdf
```

Required result categories:

```text
captured_sixel
captured_text_only
no_graphics_emitted
unsupported_document
process_failed
timeout_no_content
```

Acceptance:

- The probe reports one category per PDF/image fixture.
- If graphics are captured, it reports protocol counts and artifact path.
- If graphics are not captured, it does not fail ambiguously with only `graphics_count=0`.

This gate prevents false claims. A clean negative is acceptable for v1.

### Gate 6 — Resize behavior classification

V1 must know whether resize is supported, degraded, or untested.

Required probe behavior:

- start PTY at one size;
- capture content;
- resize PTY;
- capture content again;
- report whether terminal dimensions and child rendering changed.

Acceptance categories:

```text
resize_validated
resize_degraded
resize_unsupported
resize_untested
```

For v1, `resize_degraded` is acceptable if documented and operator-facing output is honest.

### Gate 7 — Linux parity

V1 should not be macOS-only unless explicitly scoped that way.

Required Linux command:

```bash
cd .tmp/par-term-probe
cargo test
cargo run -- --validate --assert
```

Acceptance:

- PTY smoke passes.
- Sixel capture passes.
- Sixel screenshot artifact passes.
- Bookokrat EPUB is either validated or skipped with explicit fixture/tool absence.

If Linux cannot run Bookokrat in the environment, the gate can pass with:

```text
Bookokrat skipped: binary unavailable
```

but PTY and Sixel must pass.

## Operator-facing behavior

### EPUB/text success

Example:

```text
Opened reader in embedded text mode.
Backend: par-term
Graphics: not required for EPUB text
```

### Sixel graphics artifact success

Example:

```text
Captured Sixel graphics from reader process.
Generated artifact: target/page-preview.png
Mode: artifact preview
```

### PDF/image no graphics

Example:

```text
This document did not emit Sixel graphics in the embedded backend.
PDF/image support is unavailable in Sixel V1 for this document.
Fallbacks:
- extract text if available
- generate static artifact through another renderer
- use Kitty backend when available
- open external viewer
```

### Missing Sixel support

If a backend cannot decode Sixel:

```text
Embedded graphics unavailable: Sixel decode capability missing.
Text mode remains available.
```

## Extension API acceptance

The extension-facing result must distinguish:

```rust
Opened
OpenedDegraded
ArtifactGenerated
Unavailable
Denied
Failed
```

For Sixel V1, graphical requests should not collapse into generic failure. They should return something like:

```text
Unavailable {
  missing: [graphics_decode_sixel],
  alternatives: [text_mode, artifact_export, external_viewer, kitty_backend]
}
```

or:

```text
ArtifactGenerated {
  path: ...,
  source_backend: par_term,
  graphics_protocols: [sixel]
}
```

## Test matrix

| Scenario | Command | Required for V1 | Expected |
|---|---|---:|---|
| PTY smoke | `cargo run -- --pty --assert` | yes | PASS |
| Sixel synthetic | `cargo run -- --sixel --assert` | yes | PASS |
| Sixel fixture | `cargo run -- --sixel-fixture fixtures/tiny.sixel --assert` | yes before implementation | PASS |
| Parser fixtures | `cargo run -- --parser-fixtures --assert` | yes | PASS for Kitty direct; iTerm diagnostic allowed |
| Bookokrat EPUB | `cargo run -- --bookokrat cockpit-test-assets/pride-and-prejudice.epub --assert` | yes if Bookokrat available | PASS |
| Bookokrat PDF | `cargo run -- --bookokrat-pdf cockpit-test-assets/image-page.pdf` | classification required | classified outcome |
| Screenshot artifact | `cargo run -- --sixel --screenshot target/sixel.png --assert` | yes before implementation | PNG artifact |
| Kitty matrix | `cargo run -- --kitty-matrix` | no, v2 diagnostic | diagnostic |
| iTerm route | `cargo run -- --iterm` | no, diagnostic | diagnostic |
| Linux validation | `cargo test && cargo run -- --validate --assert` | yes before v1 complete | PASS or explicit skip for Bookokrat |

## V1 done definition

Sixel V1 groundwork is complete when:

1. The probe has hard validation for PTY, Sixel, screenshot artifact, and EPUB text.
2. A real Sixel fixture exists and passes.
3. Screenshot/artifact output is generalized beyond a hardcoded path.
4. PDF/image fixtures produce classified outcomes.
5. Linux parity is run and recorded.
6. Extension-facing capability names and result variants are documented.
7. Docs state clearly that Kitty is V2 and iTerm is diagnostic.

## V1 risks

### Risk: Bookokrat may not emit Sixel

Mitigation:

- V1 still supports Sixel graphics when emitted.
- PDF/image mode returns classified unavailable/degraded results.
- Artifact fallback can use non-Bookokrat renderers later.

### Risk: artifact screenshot is useful but not interactive

Mitigation:

- Describe V1 as artifact/degraded graphics, not live graphics.
- Keep live rendering as a separate acceptance gate.

### Risk: Linux PTY/screenshot behavior differs

Mitigation:

- Run Linux validation before claiming cross-platform V1.

### Risk: Sixel ecosystem is not the modern terminal path

Mitigation:

- V1 uses Sixel because it is currently proven.
- V2 tracks Kitty as the modern path.

## V2 handoff conditions

Only move Kitty out of diagnostic status when:

- `--kitty-matrix` has at least one terminal-routed pass; or
- a direct-parser interception layer captures Kitty APC from the PTY stream; or
- an external Kitty backend is selected for live graphical mode.

Until then:

```text
Kitty direct parser support: validated
Kitty terminal-routed support: unresolved
Kitty product support: v2
```
