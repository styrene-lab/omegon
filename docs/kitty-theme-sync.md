# Kitty theme sync — gentle alpharius.conf propagation

## Overview

When the alpharius palette changes, the installed Kitty conf at ~/.config/kitty/themes/alpharius.conf goes stale. We need a way to keep it in sync without nuking user customizations. The AGENTS.md hash-guard pattern is the reference model: detect ownership, warn on drift, never silently overwrite user edits.

## Research

### AGENTS.md hash-guard pattern (reference model)

defaults.ts already implements a content-hash guard for AGENTS.md:
- Computes sha256 of the deployed content, stores the short hash at ~/.pi/agent/.agents-md-hash
- On session_start: re-read the deployed file and compare its hash to the stored one
- If hashes match: the file is Omegon-managed → safe to update
- If hashes differ: user has edited the file → warn, never overwrite
- If no hash file exists: legacy install → adopt current content as baseline (no overwrite)

The same three-way logic applies perfectly to alpharius.conf.

### Kitty conf install path and structure

- Installed path: ~/.config/kitty/themes/alpharius.conf
- Activated via: `include themes/alpharius.conf` in kitty.conf (already in user's config)
- Generated by scripts/export-kitty-theme.ts from themes/alpharius.json
- alpharius.conf has a generated-file comment: "Generated from themes/alpharius.json by scripts/export-kitty-theme.ts"
  — this is the ownership marker (analogous to the <!-- managed by Omegon --> sentinel in AGENTS.md)
- Kitty reloads config on ctrl+shift+F5 or `kitten themes` — no restart required

Unlike AGENTS.md, the Kitty conf is purely decorative/cosmetic, so the risk of silent update is lower.
But the principle holds: user may have layered custom overrides on top and we should not destroy them.

## Decisions

### Decision: Use generated-file comment as ownership marker, no separate hash file

**Status:** decided
**Rationale:** The comment "# Generated from themes/alpharius.json by scripts/export-kitty-theme.ts" already exists in the file and is unambiguous. If absent, the user wrote the file — don't touch it. A separate hash file adds state with no benefit since the conf content is deterministic from the palette.

### Decision: Write file silently, notify with reload hint — no kitten @ set-colors

**Status:** decided
**Rationale:** kitten @ set-colors requires remote control (not universal) and would affect all open windows unexpectedly. Write the file and emit a one-line pi notification: "Omegon: Kitty theme updated — reload with Ctrl+Shift+F5". User controls when the reload happens.

## Implementation Notes

### File Scope

- `extensions/defaults.ts` (modified) — Add syncKittyTheme() helper called from session_start, after OSC 10/11 emission

### Constraints

- Ownership marker: presence of '# Generated from themes/alpharius.json' in the installed file
- Four-state matrix: missing→install, current→no-op, stale+owned→update+notify, unowned→warn-once
- Never write if ownership marker absent
- Warn at most once per session (track with a module-level flag)
- Source of truth: themes/alpharius.conf in the repo (not regenerated at runtime)
