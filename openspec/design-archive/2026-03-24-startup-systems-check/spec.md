## Decisions

### GPU/VRAM detection: sysctl on macOS, nvidia-smi on Linux, tiered thresholds (decided)

macOS: `sysctl hw.memsize` gives total unified memory (shared CPU+GPU on Apple Silicon). No separate VRAM — all memory is available to Metal. Thresholds: ≥32GB = BeefyLocal (14B-32B models), ≥16GB = SmallLocal (4B-8B), <16GB = can't usefully run local models. Linux: `nvidia-smi --query-gpu=memory.total --format=csv,noheader,nounits` gives VRAM in MiB. Thresholds: ≥12GB = BeefyLocal, ≥8GB = SmallLocal, no GPU or <8GB = cloud only. These are conservative — a 14B Q4 model needs ~8-10GB, a 32B Q4 needs ~18-20GB. We want the recommended model to run comfortably, not swap.

### Yes, probe well-known ports for OpenAI-compatible endpoints (decided)

A quick GET /v1/models to localhost:11434 (Ollama), :1234 (LM Studio), :8080 (TGI/vLLM) is cheap (~50ms per port, fail-fast on connection refused). If something responds with a valid model list, surface it in the bootstrap panel and add it to the available providers. This makes Omegon work out-of-the-box with corporate inference endpoints and local tools the user already has running. Don't scan random ports — only well-known defaults. User can always add custom endpoints via env var or /login.

### Systems check drives automatic defaults AND informs the operator — both, not either/or (decided)

The system should auto-select sensible defaults (local model for compaction if GPU is beefy, OpenRouter free model for cleave children if no paid key) AND tell the operator what it chose and why in the bootstrap panel. The operator can override anything via /model, /context, settings. The cost philosophy applies: Omegon makes the productive choice by default and shows its work. Not asking permission for every routing decision — that's what tunable levers are for.
