## Decisions

### Option C: submodule init + prompt instructions + orchestrator auto-commit fallback (exploring)

Submodule init is a prerequisite (worktrees don't inherit submodule checkouts). Prompt instructions teach the child the correct procedure. Auto-commit catches failures deterministically. The three pieces are each independently useful and compose cleanly. The auto-commit runs after every child regardless — it's a no-op when the submodule is clean, and a safety net when it's dirty. Gate on `git submodule status` in the worktree to avoid unnecessary work in repos without submodules.

### Auto-commit runs always but gates on git submodule status — transparent when no submodules, active when dirty (decided)

The auto-commit calls git submodule status first. If no submodules exist, it's a no-op (~1ms). If submodules exist but are clean, it's a no-op. Only when dirty does it commit. This means it's safe to run unconditionally after every child — no scope-path gating needed. Simpler code, no false negatives.

### Three approaches compared



### Option A: Prompt engineering (tell children to commit submodules)\n**Pro**: Zero orchestrator changes. **Con**: Unreliable — children may ignore instructions, especially under token pressure. The vault-client child wrote 834 lines of correct code but failed to commit it. Adding 10 lines of instructions doesn't guarantee compliance. Also doesn't handle the case where the child crashes or times out mid-work.\n\n### Option B: Post-child auto-commit in orchestrator\n**Pro**: Transparent, deterministic, works even if child crashes. **Con**: The orchestrator makes assumptions about commit messages and what to include. Needs to distinguish intentional uncommitted files (build artifacts) from work product.\n\nImplementation: After `dispatch_child` returns, before marking as completed:\n1. `git -C <worktree>/core status --porcelain` (for each submodule)\n2. If dirty: `git -C <worktree>/core add -A && git -C <worktree>/core commit -m \"feat(<label>): <auto-generated>\"`\n3. `git -C <worktree> add core && git -C <worktree> commit -m \"chore: update core submodule\"`\n\nThis is ~30 lines in orchestrator.rs, in the post-dispatch phase.\n\n### Option C: Submodule init + hybrid prompt + auto-commit fallback\nRun `git submodule update --init` in the worktree after creation (ensuring the submodule is properly checked out). Include submodule commit instructions in the task file (Option A). Then run auto-commit as a safety net (Option B). Best of both worlds — the child learns the right procedure for next time, and the orchestrator catches failures.\n\n**Recommendation: Option C** — the init step is needed regardless (worktrees may not have initialized submodules), the prompt helps children learn, and the auto-commit is a safety net that costs nothing.
