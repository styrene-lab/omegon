# Proposal: Scenario-First Task Generation

## Intent

When OpenSpec specs are decomposed into `tasks.md` for cleave children, task groups should be organized by **spec domain** (what must be true) rather than **file layer** (where code lives). This prevents cross-cutting spec scenarios — particularly enforcement concerns like RBAC — from falling between children.

## Scope

1. **Skill instructions** — Update `skills/openspec/SKILL.md` with explicit scenario-first grouping guidance for task generation
2. **Spec-domain annotations** — Parse `<!-- specs: domain/name -->` comments from tasks.md group headers to deterministically map scenarios to children
3. **Orphan detection + auto-inject** — After scenario-to-child matching, inject unmatched scenarios into the closest child with a `⚠️ CROSS-CUTTING` marker
4. **Scenario matching rewrite** — Replace word-overlap heuristic in `buildDesignSection` with annotation-first matching, falling back to scope-based and then word-overlap

## Success Criteria

- Every spec scenario in a change's `specs/` directory appears in at least one child's acceptance criteria after matching
- Orphaned scenarios are auto-injected with markers (no silent drops)
- Task groups generated by `/opsx:ff` are organized by spec domain, not file layer
- Existing tests continue to pass; new tests cover annotation parsing, orphan detection, and auto-injection
